比较重要的一个漏洞是Time-Of-Check To Time-Of-Use漏洞

简单说，当你仅仅确认你的目标路径有文件时就能打开，就很有可能执行恶意攻击者的代码，因为操作系统是并发的，在确认到打开的这段时间，攻击者完全可以调换文件。本质上，这是属于多个实体共享一个**可变**状态时会出现的问题，例如多线程。所以会存在原子操作

模块化设计和越简单越好，以及平衡成本，安全性和易用性。

我已经理解缓冲区注入攻击的原理了

简单来说，ret是唯一会修改eip指针的地方，所以 ~~只要能到达那个地方，道路就会不断延伸~~ 只要能找到到原本的返回指针存放的地址，修改地址到自己的已知区域就行，然后执行恶意代码

而缓冲区溢出，则是通过从栈顶写入数据，而对于缓冲区又没检查的情况下就会爆栈，到达一个栈帧的栈底，也就是存放old ebp(旧的栈底位置)和old eip(调用者函数的位置，也是return address)，所以修改到这里就行

另外，关于gdb调试

常见的命令有

x /Nuf  expr  x是examine的简写，检查从address expr(可以是开了-g之后的函数名，也可以是地址本身)开始，N个u大小的内存，u指代单个读取的大小，例如w指代word，与计算机本身位数有关，b就指代单个byte

i f 查看当前被调用函数的栈帧，用来确定保存return address的位置和栈顶到return address的位置。

但是，gdb展示出来的栈空间情况并不总是真的。可能gdb自己要往栈里塞点东西用来调试，可能系统环境变量和参数的数量改变，各种原因导致栈空间的变化，导致本来放在A位置的shell code被乾坤大挪移到A+20或A-20去了。所以我们可以来搞一个nop，nop的作用只有一个：指针+1，把足够多的nop塞在shell code前面，这样即使栈空间位置整体有变化，只要A落在nop中，eip指针就会如同滑雪橇一样一直+1,直到抵达shell code的位置

另外，在bash中，写内存的方式是用printf(输入)，这样不会有别的诸如\x00截断的干扰

另外，可以不用存SHELLCODE,而是分成几段来输出

哦，还有有符号与无符号攻击

假设一个函数
```cpp
void foo(int len,char *p){
    char buf[64];
    if(len>64){
        return;
    }
    memcpy(buf,p,len);
}
```
嗯，看起来一切正常，甚至防止了缓冲区溢出

但是，我们观察memcpy的定义

```cpp
void * memcpy(void *__restrict__ __dest, const void *__restrict__ __src, size_t __n)
```

当len=-1时，补码存储情况是全1,而size_t是无符号整型，这个时候就会认为len=2^32-1，好了，又是一个缓冲区溢出

我去，还有无符号整型溢出攻击

把代码稍微修改一下

```cpp
void foo(size_t len,char *p){
    char *buf=malloc(len+2);
    if(buf==NULL){
        return;
    }
    memcpy(buf,p,len);
    buf[len]='\n';
    buf[len+1]='\0';
}
```

嗯，看起来一切都很好，

然而当len=全1时，无符号整型自动溢出，直接变为1或别的，然后失败
